# MongoDB Restore Instructions

Based on the analysis of your `infra` project, here is how your MongoDB backup and restore process works.

Your MongoDB setup uses Docker Swarm. The database runs as the `mongo-rs-1` service in the `infra` stack, configured as a single-node replica set (`rs0`), and its data is stored in the Docker volume named **`infra_mongodb-rs-1-data`**.

The `mongo-rs-backup/backup.sh` script generates a logical backup using `mongodump`, which is then compressed into a `.gz` archive and uploaded to Google Cloud Storage. Restoring it involves wiping the existing database volume to ensure a clean state, letting the replica set re-initialize, and streaming the archive into the fresh instance over the Docker overlay network.

Follow these steps on your VPS to safely restore your MongoDB data:

### 1. Locate the backup file on your VPS
Ensure your backup file (e.g., `mongodump-YYYY-MM-DD_HH-MM.gz`) is present on the VPS. 
*For the commands below, assume it is located in the current directory (`$(pwd)`) and is named `mongodump.gz`.*

### 2. Stop dependent services
Before restoring, it is highly recommended to stop any services that connect to MongoDB to prevent conflicts or partial writes while you reset the database.
```bash
docker service scale infra_YOUR_SERVICE=0
```

### 3. Wipe the existing database volume
To ensure a clean restore without duplicate key errors, you should start with a fresh database. We will stop Mongo, remove its volume, and start it again.
```bash
# Stop the MongoDB service
docker service scale infra_mongo-rs-1=0

# Wait a few seconds for the container to stop completely, then remove the volume
docker volume rm infra_mongodb-rs-1-data

# Start the MongoDB service back up
docker service scale infra_mongo-rs-1=1
```

### 4. Re-initialize the Replica Set
Because we wiped the volume, the replica set configuration is lost. We need to trigger the initialization script again.
```bash
# Force the initialization service to run again
docker service update --force infra_mongo-rs-init
```
> **Note:** Wait for about 15-30 seconds to ensure MongoDB has fully initialized the replica set and is ready to accept connections. You can check the logs with `docker service logs infra_mongo-rs-init`.

### 5. Restore the database from the archive
Your backup is a gzip-compressed archive generated by `mongodump`. We will use a temporary Mongo container attached to the `infra_mongo-rs-net` overlay network to restore the archive into the newly running Mongo instance.

Run the following command from the directory where your backup is located:
```bash
docker run --rm -i \
  --network infra_mongo-rs-net \
  mongo:8.2.3 \
  mongorestore \
  --uri="mongodb://YOUR_MONGO_USER:YOUR_MONGO_PASSWORD@mongo-rs-1:27017/?replicaSet=rs0&authSource=admin" \
  --archive --gzip --drop < mongodump.gz
```
> **Note:** Replace `YOUR_MONGO_USER` and `YOUR_MONGO_PASSWORD` with your actual MongoDB credentials (often defined as `${USERNAME}` and `${PASSWORD}` in your `.env` file). The `--drop` flag ensures that any existing collections are dropped before restoring. Using `--archive` without a file path tells `mongorestore` to read from standard input.

### 6. Restart dependent services
Once the restore finishes, start your dependent services back up:
```bash
docker service scale infra_YOUR_SERVICE=1
```

Your MongoDB collections and data should now be fully restored!
